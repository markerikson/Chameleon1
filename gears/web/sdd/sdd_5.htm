5.2 Architecture

5.2.1 System Architectural Components

I'm looking for an architectural diagram that shows the major top-level components for the project. Typically, there might be about 5 +/- 2 shown here, but not more than 10. Be sure to keep the diagram simple, minimizing connection lines drawn between the components (typically those connection lines indicate communication coupling). Quite often the most common top-level architectural components designed by students are (1) the graphical user interface, (2) some kind of a computation engine, (3) maybe another computation engine, and sometimes (4) some kind of HTML output report generator.
--JC.

5.2.2 Modules in Components

I'm looking for a list of all the modules (or classes) that comprise each high-level architectural component. Please note that there may be anywhere from one to many modules that comprise one of the high-level architectural components. Be sure that all of the modules are listed here (number each one), categorizing them according to the components that they make up. Each one of these modules will be listed and described in great detail later in section 5.9. Here's a little example, based on the "SumDiv" sample project I have available that demonstrates how to build a project using VB and C++:
--JC.

Architectural Component #1:
SumDiv Calculator Engine (C-based DLL)

  1. Initialization module (e.g. DLL main)
  2. Addition module
  3. Division module

Architectural Component #2:
SumDiv Graphical User Interface (VB-based EXE)

  1. Initialization class (e.g. form load, etc.)
  2. LHS Value (text box class)
  3. RHS Value (text box class)
  4. Addition Command (command button class)
  5. Division Command (command button class)
  6. Results Value (text box class)

5.3 Design Constraints

List any design constraints or assumptions that your design will be based on.
--JC.

5.4 Global Data Objects

List any global variables that will be designed into your system (hopefully there wont be any). Be sure to explain why these global data objects are designed to be global. A common (but poor) example of this is project that has hard-coded output file names that the user cannot dictate at run-time. Somewhere within the project is a global variable or constant that specifies this file name.
--JC.

5.5 Error Handling

Explain how you plan to handle internal, user, and system errors. For example, if you have internal errors, will the assert macro be used liberally throughout your source code to detect such an item? Or if the user enters invalid data, do you have a consistent and user-friendly way to handle this for all cases? If your application is running and some part of the system goes down (e.g. the user pressed Ctl-Alt-Del), do you have a way to recover the work-in-progress file(s) from such an abrupt abort? Discuss these and any other related items in this section.
--JC.

5.6 Development Language

Describe the language that will be used for development of the software. For example, is it ANSI C, or K&R C, etc. What brands and versions of compilers and linkers will be used to ensure consistency?
--JC.

5.7 Preconditions and Postconditions

List any and all preconditions that must be met prior to starting your application. For exmaple, if your program relies on specific databases for initialization, then so state. Does your application rely on any system DLLs for execution?
--JC.

Describe any and all postconditions that shall be guaranteed by the system. For example, if one of your modules opens a data file for writing, will that same module be responsible for closing the data file, too? When will such a data file be closed?
--JC.
