4.0 Interface Control Document (ICD)

The ICD describes all the source code interfaces used between all various components of the system. It also helps isolate the component design and development, and promotes functional-level stubbing/testing. Plus it forces thinking in terms of the data flow rather than processes. What needs to be described in this section are all the functional software routine interfaces used in the system between modules within the Architectural Components (the AC's are documented in the next section of the document, but should also be referenced here for completeness).

When you enter the implementation phase of the software lifecycle, you'll essentially take everything documented in the ICD and convert it to C-based header files. Here's an example for one AC, but you'll need to do this for all the AC's in your system:

4.1 "BLDINFO" (Build information and details)

The following declares the constants, data structures, and functional interfaces exported for use by other clients of this module.

----------------------------------------------------------
 GetBuildLevel() returns the numeric value of the build number.
BLDINFO_API  unsigned long  _stdcall  GetBuildLevel(void);


----------------------------------------------------------
 GetBuildDateAndTime() returns the time details about the build.
 If the function returns TRUE, then the return parameters 
 contain valid data.  If the function returns FALSE, then the
 function could not get the build data and time data.
BLDINFO_API  BOOL  _stdcall  GetBuildDateAndTime(
   unsigned short * minutes,
   unsigned short * hour,
   unsigned short * day,
   unsigned short * month,
   unsigned short * year);
