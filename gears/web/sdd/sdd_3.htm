3.0 Naming and Coding Standards

3.1 Naming Standards

Provide a description of the naming standards that your team will use for the design and implementation of the software. This section should include the rules that you and your team agree upon regarding any of the following:

  • File Name Styles (e.g. InitialzeTheSystem.c versus SysInit.c)

  • File Name Length (e.g. long file names versus DOS's 8+3 style)

  • Function Names (e.g. is each name a Verb-Noun style like in the DFD?)

  • Header Names

  • Global Data names

  • Local Data names

  • Hungarian Naming Convention? (see http://www.freenet.tlh.fl.us/~joeo/hungarian.html or also http://www.csci.csusb.edu/doc/hungarian)

We have no particular naming standards at this time, mostly because each member of the team is responsible for their own code files.

3.2 Coding Standards

Provide a detailed description here of the coding standards that your team agrees on. For example, you may want to have your team follow something like this:

  • Each function shall have a prototype.

  • Each function prototype shall be listed in a header file.

  • Only source code files (*.cpp files, not *.h files) shall include header files.

  • Functions shall not exceed 100 lines.

  • Each variable shall have a comment describing the variable's purpose.

  • A consistent indentation of 3 spaces shall be used for each block.

  • Comments shall be written at the level of intent (say why and what, not how).

  • Each function shall have a header block describing the function name, purpose, each argument (input or output and purpose), the function's return value, side effects, and assumptions.

  • Each file shall have a modification history (revision log) showing the version number, date, person, and description of each change.

F eel free to add/modify/delete items on this list, depending on your group's personal desires.
--JC.
We have no particular coding standards at this time, for the same reason that we have no particular naming standards.
