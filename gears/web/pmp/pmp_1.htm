1.0 Introduction

1.1 Project Overview

This document describes the management processes for Digital Gearheads' senior design project, an adaptable IDE codenamed "Chameleon". The project's main objectives are to learn about software engineering techniques and ultimately produce a working program usable by beginning programming students. There are several milestones for this project. By November 20, a basic rough version of Chameleon should exist. Further major milestones, such as the end of the fall semester, will be worked out as project progress becomes clearer. In between, each two-week iteration will end in a release.  At the end of the spring semester, Chameleon should be complete.  Our resources are Mark's all-nighters, Ben's cunning, Dave's Linux server, and a Holocaust cloak.  Our budget is about 42 cents.

Chameleon's details can be seen in the Software Requirements Specification document.

1.2 Project Deliverables

Deliverables for the Chameleon project include the documentation (the Software Configuration Management Plan, the Project Management Plan, the Software Requirements Specification, and the Software Design Document), the software source code, program executables, and whatever actual program documentation or tutorials that is decided upon later.

1.3 Evolution of the PMP

This document is currently an initial draft.  It will be reviewed as needed during team meetings and updated as necessary.  Project requirements will be updated through discussion with the customer as progress is made.  The project schedule will be updated after discussion at team meetings.

1.5 Definitions and Acronyms

Activity
A major unit of work with dates for begin and end (e.g. documenting the plans used for configuration management, or designing the project, or developing a rapid prototype, etc.).
Milestone
The date when a Work Product is completed (e.g. the date when the SRS is submitted and is totally done).
PMP
Project Management Plan.
Task
This is a sub-component of an activity (e.g. completing the first draft of the SRS).
Work Package
A textual description of the work product, the required staffing and resources, the duration, name of person responsible for the work product, and acceptance criteria.
Work Product
The results of an activity (e.g. the SDD is the result of the Design phase, the SRS is the result of the Requirements Specification phase, the schedule and budget are the results of the planning activity, etc.).
