4.0 Test Cases

Simply state that this section lists all of the test cases.

When writing each test case, assume that the reader (tester) is a customer. This means that the customer will only know about the whole product, not individual modules (i.e. don't say things like "the GUI module will pass an intermediate file to the Engine.DLL", because the customer has no clue what GUI or Engine means, nor this intermediate file). Also be sure that if you need to use a certain file for a test case, then that file needs to be prepared ahead of time (store it in the "tests" subdirectory), and the test case needs to explicitly direct the reader to load this file.
--JC.

4.n Test Case n

4.n.1 Purpose

Describe the purpose of this test case. Usually this includes some mention of either verifying a requirement and/or ensuring that a given logic path is traversed.
--JC.

4.n.2 SRS Requirement

Describe the SRS requirement for this test case. Provide a hyperlink to the specific SRS requirement.
--JC.

4.n.3 Environmental Needs

Describe any and all environmental needs for this test case (e.g. Internet connection, operating system, printer connection, etc.).
--JC.

4.n.4 Inter-case dependencies

Describe any and all inter-case dependencies for this test case (e.g. test case 4.6 will only work if test case 4.2 was successful). It might be helpful to provide a hyperlink to any dependent test cases.
--JC.

4.n.5 Special Procedure Requirements

Describe any and all special procedure requirements for this test case (e.g. before running this test case, may need to manually edit some input file to inject an error to test error handling, etc.).

For example, if a specific file used in this test case needs to be copied from your "stp/tests" subdirectory to the "C:\TEMP" directory on the tester's machine, then explain that here like this:
--JC.

Copy the file test01.txt to "C:\TEMP\TEST01.TXT".

Sometimes this section is not applicable, but it is provided here in case you need it.
--JC.

4.n.6 Procedural Steps

In painstaking detail describe the procedural steps that a user must do to perform this test case. Write the instructions assuming that the tester is a customer and knows nothing about internal data files, DLL interaction, or any other developer-known secrets. Here's an example:
--JC.
  1. If the application is currently running, close it.
  2. Invoke the application.
  3. Open an existing data file for the application to use (e.g. go to File | Open). In the Open File dialog box navigate to the file named "C:\TEMP\TEST01.TXT" and click the Open button.

    Expected Results: Before continuing this test case, verify that the application loads the specified data file successfully. (If it fails to load properly, a warning message would be displayed.)

  4. Validate the loaded data file (e.g. go to Tools | Validate).

    Expected Results: Before continuing this test case, verify that the application displays a message box that looks like this:

  5. Click the OK button to dismiss the message box.
  6. Exit the application (e.g. go to File | Exit).

4.n.7 Pass/Fail Criteria

Normally you would say that if the expected results match the actual results, then the test case passes; otherwise it fails.
--JC.
