4.3 Configuration Control

We will be using various CVS clients, depending on needs and current working platform. Under Windows, this may include WinCVS and TortoiseCVS; under Linux, this may include Cervisia, and the command-line CVS client.

We will be maintaining one SCML, using CVS. To this, the SCM Users will commit all code changes. At the end of each iteration, after advisement from the Test/QA Manager, the SCM Administrator will cvs-tag the code as a stable build release.

CVS provides the ability to record comments when making a checkin. Each checkin has its time and author listed as well. This will serve as our main means of tracking changes.

We intend to maintain a policy of individual code ownership. This should minimize conflicts involving multiple developers working on the same code. Still, CVS is designed to handle multiple users working on the same file simultaneously, and is capable of intelligently merging file differences. Should more than one developer be working on a file at the same time, problems should be minimal.

4.4 Configuration Reviews/Audits

The SCM Administrator will monitor the SCML after each check-in/creation, and alert the team to problems that may have been caused by the changes.

4.5 SCML Archives

CVS is designed to do archiving, so no additional archiving will be done.

Because of the structure of cvs usage, every developer will have a working copy of the SCML in their development environment. As a result, no explicit form of backup will be used.
